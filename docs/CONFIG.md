# YAML configuration reference

## `multicast_pubsub:` — top-level component

```yaml
multicast_pubsub:
  id: my_pubsub                 # optional, only needed if you have multiple instances
  port: 18512                   # uint16, default 18512
  scope: link-local             # link-local | site-local | organization-local
  hops: 1                       # uint8 1..255, default 1
  on_message:                   # list of topic-specific triggers
    - topic: "..."
      then:
        - <any automation actions>
```

| Key          | Required | Type     | Default       | Description                                                                |
|--------------|:--------:|----------|---------------|----------------------------------------------------------------------------|
| `id`         | no       | id       | autogenerated | Reference id if you need to address this instance from another component.  |
| `port`       | no       | uint16   | `18512`       | UDP port. Both publishers and subscribers must use the same value. Adjacent to ESPHome's `udp:` default (18511). |
| `scope`      | no       | enum     | `link-local`  | IPv6 multicast scope. See `docs/PROTOCOL.md` §2.1.                          |
| `hops`       | no       | int 1–255| `1`           | `IPV6_MULTICAST_HOPS`. Raise above 1 if you need multi-subnet routing.      |
| `on_message` | no       | list     | empty         | Topic-keyed triggers. The trigger argument `x` is `std::vector<uint8_t>`.   |

The component is `MULTI_CONF`-friendly: you can declare two
`multicast_pubsub:` blocks with different `port:` values to keep separate
fabrics on the same device.

## `multicast_pubsub.publish:` — action

```yaml
on_...:
  - multicast_pubsub.publish:
      id: my_pubsub              # optional, only with multiple instances
      topic: "home/temp"
      payload: !lambda 'return std::to_string(x);'
```

| Key       | Required | Type             | Notes                                                                                  |
|-----------|:--------:|------------------|----------------------------------------------------------------------------------------|
| `id`      | no       | id               | Pick a specific `multicast_pubsub:` instance.                                          |
| `topic`   | yes      | templatable str  | UTF-8, non-empty, ≤200 bytes, no NUL.                                                  |
| `payload` | yes      | templatable str  | Raw bytes-as-string. Maximum 496 bytes; literals over the limit are rejected at config time. |

If `payload` is a runtime lambda we can't size-check at config time. An
oversize payload at runtime is logged at `ERROR` level and the publish call
silently fails (returns false in C++).

## `sensor:` platform: `multicast_pubsub`

```yaml
sensor:
  - platform: multicast_pubsub
    multicast_pubsub_id: my_pubsub   # optional
    topic: "home/livingroom/temp"
    name: "Living Room Temperature"
    mode: subscribe                  # subscribe | publish | both
    # plus any standard sensor schema knobs
    unit_of_measurement: "°C"
    accuracy_decimals: 1
    filters:
      - filter_out: nan
```

| Key                   | Required | Type   | Default     | Notes                                                            |
|-----------------------|:--------:|--------|-------------|------------------------------------------------------------------|
| `multicast_pubsub_id` | no       | id     | auto        | Which `multicast_pubsub:` instance to attach to.                 |
| `topic`               | yes      | str    | —           | Topic name, same constraints as the action.                      |
| `mode`                | no       | enum   | `subscribe` | `subscribe` (decode payload as float → sensor state) / `publish` (sensor state → ASCII float on the wire) / `both`. |

All other keys come from the standard ESPHome
[`sensor.sensor_schema`](https://esphome.io/components/sensor/) — units,
filters, device class, `on_value`, MQTT discovery, etc., all work normally.

## Topic naming rules

* UTF-8, non-empty, no NUL bytes.
* Maximum 200 bytes encoded.
* Case-sensitive. `Home/Temp` ≠ `home/temp`.
* No wildcard support — each subscription must be an exact topic.
* MQTT-style `/`-separated hierarchies are conventional but not required.

## Coexisting with MQTT

The component does not conflict with ESPHome's `mqtt:` component; both can
appear in the same YAML and share `on_message` triggers + `publish`
actions. See [`examples/05_mqtt_bridge.yaml`](../examples/05_mqtt_bridge.yaml).
The C++ MQTT client only builds for ESP targets (it uses
`AsyncMqttClient-esphome`), so bridge configurations target real hardware
rather than `platform: host`.
